Pipeline Ventas: de CSV mensual a mart analítico, gobernado por metadata

Pipeline end-to-end que ingiere entregas mensuales de CSV (TPC-H) en Snowflake, las historiza como Data Vault 2.0 con dbt y las expone en una estrella Kimball, orquestado por Airflow. Todo el pipeline se parametriza con una sola variable (load_id) y se define en metadata declarativa (YAML).

2 cargas: 202512 · 202601
1 parámetro: load_id
41 modelos dbt

El enfoque: metadata-driven de verdad

Un contrato YAML por entidad (metadata/entities/*.yml) es la única fuente de verdad: origen del fichero, columnas con tipos, tipo de carga (full/append), clasificación Data Vault (hub o link, clave de negocio) y dependencias de orquestación. De ese único fichero se derivan el CREATE TABLE de Bronze, el COPY INTO, las vistas de staging con sus hashes y las tareas de Airflow — lo consumen tres piezas independientes: el motor de ingesta, el generador de código dbt y la fábrica de DAGs.

La metadata se valida contra un JSON Schema (metadata/schema/entity.schema.json) con pytest, en segundos y sin tocar Snowflake: la metadata es código, y se valida como código.

"Si mañana llega una novena tabla, es un fichero YAML nuevo — cero líneas de Python o SQL tocadas."

Arquitectura por capas — Medallion real, no solo narrativa

Cada capa es una base de datos de Snowflake, no un esquema: GRANT ON DATABASE es la frontera de aislamiento más simple de razonar y la más difícil de traspasar por error.

Ingesta: un único motor genérico e idempotente

ingestion/loader.py no contiene lógica de ninguna entidad concreta. Por cada entidad ejecuta el mismo contrato de 5 pasos:

  • 1 · Crear si no existe — CREATE TRANSIENT TABLE IF NOT EXISTS en Bronze, con las columnas del YAML.
  • 2 · Subir el fichero — PUT al stage interno STG_VENTAS_CSV, organizado por {load_id}/{entidad}/.
  • 3 · Limpiar la partición — TRUNCATE (entidades full) o DELETE WHERE load_id (append): relanzar nunca duplica.
  • 4 · Cargar — COPY INTO con FORCE=TRUE siempre: el load history de Snowflake sobrevive al DELETE y, sin FORCE, un re-run saltaría los ficheros ya vistos cargando 0 filas en silencio.
  • 5 · Auditar — INSERT en META.CONTROL.TB_VENTAS_LOAD_AUDIT con estado SUCCESS/FAILED, filas cargadas y run de Airflow.

De las 8 entidades, 6 son full (catálogos y maestros) y 2 son append (orders, lineitem: los hechos transaccionales que crecen cada mes).

Silver: Data Vault 2.0 — historia sin UPDATE ni DELETE

20 objetos generados desde la misma metadata que alimenta la ingesta: 6 hubs (una fila por clave de negocio), 6 links (relaciones) y 8 satélites (atributos historizados por hashdiff).

Dos hashes con propósitos distintos: la hash key (hk_) identifica — MD5 de la clave de negocio —, y el hashdiff (hd_) detecta cambio: huella de todos los atributos, comparada contra la versión anterior sin mirar columna a columna.

El SQL repetitivo del patrón (deduplicación con ROW_NUMBER, historización con LAG sobre hashdiff) lo genera el paquete automate_dv: se escribe una vez, no veinte.

Los dos matices de modelado que definen el diseño:

  • lineitem no tiene hub: es una entidad débil (no existe fuera de su pedido), modelada como link de order+part+supplier con l_linenumber como dependent child key.
  • partsupp es el link + satélite de libro: el stock y el coste pertenecen a la relación pieza-proveedor, no a ninguna de las dos por separado.

El vault es insert-only: nada se actualiza ni se borra; si el hashdiff no cambió, no se inserta nada. Esa propiedad es la segunda mitad de la idempotencia del pipeline.

Orquestación: el DAG se construye solo desde la metadata

dags/dag_factory.py es el único fichero de DAGs y no menciona ninguna tabla ni entidad: en cada parseo lee la metadata y construye el DAG ventas_monthly con esta secuencia:

ingest (8 tasks, cableadas por depends_on) → dbt_raw → dbt_staging → dbt_raw_vault → dbt_mart → verify

  • astronomer-cosmos traduce cada modelo dbt en una task nativa de Airflow (selección por tag:<capa>): el hub/link/satélite se ve dibujándose solo en la UI, con estado y logs por modelo.
  • dbt vive en un venv aislado dentro del contenedor (conflicto real de protobuf entre dbt-core y Airflow); Cosmos lo invoca por subprocess apuntando al binario del venv — el patrón que recomienda Astronomer.
  • verify cierra el pipeline: falla el run si alguna entidad de este (load_id, run) no tiene SUCCESS en la tabla de auditoría.

Disparar el mismo DAG cambiando solo {"load_id": "202601"} resuelve el segundo drop completo — ni un fichero tocado, ni una línea de código. Números verificados en Snowflake:

Coste bajo control

Un warehouse XS por carga de trabajo — WH_VENTAS_INGEST_XS, WH_VENTAS_DBT_XS, WH_VENTAS_BI_XS — todos con auto-suspend a 60 s y auto-resume: la ingesta nunca compite con BI, y escalar es un ALTER, no una migración.

Calidad en tres frentes

  • Contrato — python -m pytest (20/20 PASS) valida cada YAML contra el JSON Schema, en segundos y sin Snowflake.
  • Datos — 80 tests dbt (unique, not_null, relationships) intercalados en el build (121/121 PASS entre modelos y tests): un modelo roto corta el pipeline antes de propagar el error.
  • Operación — la task verify + META.CONTROL.TB_VENTAS_LOAD_AUDIT trazan cada carga de punta a punta.

Camino a producción: Azure + CI/CD

Cada pieza local tiene un equivalente productivo directo — la lógica (metadata, loader, dbt, DAGs) no cambia; solo cambia dónde viven los ficheros, las credenciales y el runtime.

"Desplegar un pipeline nuevo = mergear un YAML."

Qué NO se construyó, a propósito

Snowpipe (la cadencia es mensual, no continua), políticas de PII (TPC-H no tiene dato sensible real) y alerting a Teams/Slack (callbacks de Airflow como punto de enganche). Decisiones de alcance, no carencias: cada una tiene un punto de enganche claro en el diseño.
