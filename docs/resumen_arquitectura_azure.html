Diseño Snowflake

4 bases de datos — una por capa

  • BRONZE · LANDING — aterrizaje crudo + columnas técnicas de auditoría.
  • SILVER · STAGING + VAULT — Data Vault 2.0: el sistema de registro.
  • GOLD · MART — estrella Kimball, la verdad de consumo.
  • META · CONTROL — control-plane transversal (auditoría de cargas).
Por qué a nivel de base de datos y no de esquema: GRANT ON DATABASE es la frontera de aislamiento más simple de razonar y la más difícil de traspasar por error. En producción: sufijo de entorno (_DEV/_PROD).

3 roles funcionales · 3 warehouses XS

LOADER_FRescribe BRONZE · inserta auditoría en META · usa WH_INGEST
TRANSFORMER_FRlee BRONZE · escribe SILVER y GOLD · lee META · usa WH_DBT
ANALYST_FRsolo lee GOLD — no ve el resto · usa WH_BI
Por qué separar warehouses: una transformación pesada nunca frena a BI, y cada uno escala según su demanda real (escalar = un ALTER, no una migración). Auto-suspend a 60 s: no se paga lo que no se usa.

Ingesta y orquestación metadata-driven

Ingesta idempotente

  • COPY INTO desde el external stage con FORCE=TRUE, precedido de TRUNCATE (full) o DELETE por load_id (append).
  • Relanzar una carga no duplica datos — verificado end-to-end.
  • Cada fichero deja su fila de auditoría en META.CONTROL.
  • Cadencia mensual → no se necesita Snowpipe; si fuera continua, solo cambia el disparador, no el contrato.

Los DAGs no se escriben: se generan

  • Un único dag_factory.py lee los YAML de metadata/ y produce el DAG completo: ingesta → capas dbt → verificación.
  • Todo se parametriza con una sola variable: load_id. Mes nuevo = mismo DAG, otro parámetro.
  • Entidad nueva = añadir un YAML, cero código.

El papel de dbt

  • Todo el ciclo de transformación ocurre dentro de Snowflake (ELT): raw → staging (hashes) → Data Vault (hubs/links/satélites incrementales, insert-only) → mart dimensional.
  • Las macros de automate_dv generan el SQL de cada tabla del vault a partir de la metadata — cero SQL a mano por tabla.
  • Sus tests (unique, not_null, relaciones) son la barrera de calidad: dbt build construye y valida en orden de dependencias, y para si algo falla.
  • Integración: Airflow (Cosmos) traduce cada modelo dbt a una tarea nativa del DAG — reintentos y logs por modelo; credenciales desde Key Vault; escribe con el rol TRANSFORMER.
